Theory & formal foundations

Companion notes for the Regular Expression Tool: languages, syntax trees, automata, minimization, and exact equivalence. Sections build from notation to algorithms you see animated in the app.

Languages & alphabets

Sets of strings

Basic vocabulary

  • An alphabet Σ is a finite, non-empty set of symbols (e.g. Σ = {a, b}).
  • A string (or word) is a finite sequence of symbols from Σ, including the empty string ε of length 0.
  • Σ* is the set of all finite strings over Σ, including ε.
  • A language L is any subset L ⊆ Σ*. It may be finite, infinite, empty ∅, or {ε}.

A regular language is any language that can be built from ∅, {ε}, and single-symbol languages using union, concatenation, and Kleene star finitely many times. Every regular language is accepted by some finite automaton; that is the deep fact behind regex engines.

Operations on languages

For L1, L2 ⊆ Σ*:

  • Union: L1 ∪ L2 = { w | w ∈ L1 or w ∈ L2 } — matches the | operator when L1, L2 are the languages of sub-expressions.
  • Concatenation: L1L2 = { xy | x ∈ L1, y ∈ L2 } — matches implicit or explicit concatenation in a regex.
  • Kleene star: L* = { ε } ∪ L ∪ LL ∪ LLL ∪ … — zero or more concatenated pieces from L.

Regular expressions

Syntax → language

Inductive reading (how theorists define them)

Over a fixed alphabet Σ, each regex r denotes a language L(r). Typical base and inductive clauses:

  • ∅ denotes ∅; ε denotes {ε}; for a ∈ Σ, a denotes {a}.
  • If r1, r2 are regexes, r1|r2 denotes L(r1) ∪ L(r2).
  • r1r2 (concatenation) denotes L(r1)L(r2).
  • r* denotes L(r)*.

Your app adds shorthands + and ?; they expand to combinations of union, concat, and star (e.g. r+ ≡ rr*, r? ≡ r|ε at the language level).

Operators in this project

  • | — union of two sub-expressions.
  • * — star binds to the shortest legal atom on its left (literal or parenthesized group).
  • + — one or more repetitions of the underlying language.
  • ? — optional (zero or one).
  • () — grouping for both semantics and readability.
  • Implicit concatenation is inserted between atoms where the grammar requires it (same role as an explicit “·” in formal writing).
Precedence, associativity, and pitfalls
  • Star, plus, and question mark bind tightly; union usually binds looser than concatenation — so ab|cd is (ab)|(cd), not a(b|c)d unless you parenthesize.
  • Union is associative; concatenation is associative at the language level (string grouping differs only by how you parenthesize the same underlying word).
  • Two regex texts can differ while L(r1) = L(r2); equivalence in the app checks language equality via automata, not string equality of patterns.
  • Whitespace in this tool is literal if present inside the pattern; there is no “free” whitespace ignore unless you treat space as a symbol in Σ.

Algebraic laws (intuition)

Same language

These identities hold for languages of regexes (over the same alphabet). They explain many “obvious” simplifications.

  • Idempotent union: L ∪ L = L; in regex text, r|r accepts the same strings as r.
  • Star absorption: L*L* = L* when L already allows arbitrary repetition — e.g. (a|b)*(a|b)* and (a|b)*.
  • Annihilator: ∅L = L∅ = ∅; no string can be built if one side is impossible.
  • Identity for concat: {ε}L = L{ε} = L.
  • Distributivity hints: L(L1 ∪ L2) = LL1 ∪ LL2; (L1 ∪ L2)L = L1L ∪ L2L.
Algebraic laws explain quick sanity checks; they do not replace a formal proof for every pair of expressions — the product-automaton test in the app is decisive for the compiled DFAs.

Parse trees

Syntax

Role of the tree

A parse tree records how the parser grouped your input: leaves are literals or ε; internal nodes are operators (union, concat, star, etc.). The tree removes ambiguity about “what the star covers” and “which operands the union combines.”

  • Every valid regex in the grammar corresponds to at least one tree; parenthesization fixes the shape when the grammar would otherwise be ambiguous.
  • A post-order traversal yields postfix notation — the same order a stack-based Thompson builder consumes.
  • The tree view in the app is a concrete printout of that structure; mismatches between what you meant and what you wrote usually show up as unexpected tree shape before you even look at the DFA.

From tree to automaton fragments

  • Each subtree corresponds to a sub-NFA with a distinguished start and accept “port.”
  • Internal nodes combine children using ε-transitions (Thompson’s recipe).
  • Because combinations are local, the overall NFA size is linear in the size of the regex for standard constructions.

Regex → NFA → DFA (pipeline)

This app

Thompson’s construction (sketch)

  • Literal: one transition on that symbol between fresh start and accept states.
  • Concatenation: ε from first fragment’s accept to second fragment’s start; new fragment spans first start to second accept.
  • Union: new start with ε to both sub-starts; both sub-accepts ε-merge into a new accept.
  • Star: new start with ε to sub-start and directly to new accept; sub-accept ε-loops back to sub-start and ε-goes to new accept.

The NFA may have many ε-edges and, for some symbols, multiple transitions from a state — that is why it is “nondeterministic.”

Why go to a DFA?

  • Simulation on an NFA without conversion can require tracking a set of states anyway (powerset thinking).
  • A complete DFA gives exactly one successor per (state, symbol) — easy stepping and visualization.
  • Subset construction makes that set-tracking explicit as DFA states.
|QDFA| ≤ 2|QNFA|

Exponential worst case; many subsets are unreachable so the built graph is often much smaller.

NFA (formal)

Nondeterminism
N = (Q, Σ, δ, q0, F)
  • δ : Q × (Σ ∪ {ε}) → P(Q) — image is a set of possible next states (P(Q) is the power set).
  • Acceptance: some computation path reads the whole input and ends in F (including any ε-moves between symbols).

ε-closure (algorithmic)

Given a set S ⊆ Q of NFA states:

  • Initialize closure ← S and a stack with all states in S.
  • While stack nonempty: pop state q; for each q′ reachable from q by a single ε-edge, if q′ ∉ closure, add q′ and push q′.
  • The result is the ε-closure of S — all states reachable without consuming an input symbol.

Subset construction uses ε-closure after every symbol move so each DFA state encodes “where the NFA could be, including silent jumps.”

Subset construction (NFA → DFA)

Algorithm

State space

Each DFA state is a subset T ⊆ Q of NFA states (often written [T] in lectures). The start DFA state is ε-closure({q0}).

Transition rule

For DFA state T and symbol a ∈ Σ:

  • Let U = ⋃q ∈ T δ(q, a) (all NFA targets on a from states in T).
  • Define δDFA(T, a) = ε-closure(U).
  • If U is empty, the move is undefined unless you add an explicit sink (dead) state; the app’s visualization follows the partial graph from reachable subsets.

Accepting condition

A subset state T is accepting for the pattern if T ∩ F ≠ ∅ — at least one NFA accept state is “currently possible.”

Worklist view: start with the start subset; repeatedly pick an unprocessed (T, a), compute δDFA(T, a), enqueue if new — exactly the BFS you see logged during construction.

DFA definition

5-tuple

A deterministic finite automaton has at most one next state per current state and input symbol (total DFAs include a sink so δ is total). It accepts w if the unique run on w ends in F.

M = (Q, Σ, δ, q0, F)
  • Q — finite states.
  • Σ — finite alphabet.
  • δ : Q × Σ → Q — transition function (partial δ corresponds to “crash” or implicit reject).
  • q0 — start.
  • F ⊆ Q — accept states.

States (Q)

Summary of progress through the pattern; each state encodes a distinguishable “memory” of the input prefix.

Alphabet (Σ)

Symbols read one per step; multi-character “tokens” are not built in unless you model them as single symbols.

δ

Deterministic step: at most one successor per (q, a). Visual graphs draw one labeled edge per symbol out of each state.

q0 and F

Start before reading; accept after consuming the entire string (no trailing input left).

Language L(M)

L(M) = { w ∈ Σ* | δ̂(q0, w) ∈ F } using the extended transition δ̂ defined below.

Simulation & extended δ̂

Running M

Extended transition δ̂

Define δ̂(q, ε) = q, and inductively δ̂(q, wa) = δ(δ̂(q, w), a) when the right-hand δ is defined.

  • String w is accepted iff δ̂(q0, w) ∈ F.
  • If some prefix leaves δ undefined, the machine rejects w (or you treat missing edges as transitions to a sink reject state).
  • The DFA tab’s stepper implements this walk — each click applies one symbol.

Time per input

On a fixed DFA, testing a string of length n takes O(n) time and O(1) space beyond the input — one of the reasons compilers and matchers like DFAs after preprocessing the pattern.

DFA minimization

Partitions

Goal

Merge states that are indistinguishable: for every continuation w ∈ Σ*, both states agree on accept/reject. The quotient machine has one state per equivalence class.

Partition refinement (Moore-style)

  • Start with partition {F, Q \ F} (accept vs non-accept), refined so the start’s block is convenient for display.
  • Refine: split a block if two states inside it disagree on which block δ sends them to for some symbol a ∈ Σ.
  • Repeat until no split happens — blocks are the minimal states.
  • Transitions between blocks are well-defined because all states in a block agree on each symbol’s target block.

This matches the high-level behavior of the minimizer in the project (split by signature of “which block each symbol reaches”).

Table-filling (alternative view)
  • Mark every (p, q) where p ∈ F and q ∉ F (or symmetrically) as distinguishable.
  • Repeat: if some unmarked pair (p, q) has a symbol a such that (δ(p,a), δ(q,a)) is marked, mark (p, q).
  • Stop when a full pass adds no marks; unmarked pairs are mergeable.
  • Same equivalence relation as partition refinement — two algorithms, one concept.
For a complete DFA over Σ, the minimal DFA is unique up to isomorphism (renaming states). That uniqueness is why comparing minimized machines is a standard exact test for language equality.

Myhill–Nerode (conceptual backbone)

Equivalence classes

Define x ~L y iff ∀w ∈ Σ*, (xw ∈ L ⇔ yw ∈ L). Classes of ~L are the coarsest memory any DFA for L must keep.

  • L is regular iff ~L has finite index (finitely many classes).
  • The number of classes equals the number of states in the unique minimal complete DFA for L.
  • Minimization finds those classes algorithmically without naming ~L explicitly.

Language equivalence of two DFAs

Product

Given M1, M2 (complete over the same Σ for simplicity), L(M1) = L(M2) iff no reachable pair (q1, q2) splits acceptance.

  • Initialize queue with (q0,1, q0,2) and a visited set of pairs.
  • Step: dequeue (p, r). If (p ∈ F1) XOR (r ∈ F2), return “different languages.”
  • For each a ∈ Σ, enqueue (δ1(p,a), δ2(r,a)) if not visited.
  • If queue empties without mismatch, languages are equal.

This is BFS/DFS on the product graph; it checks language equality, not graph isomorphism and not equality of state IDs.

Bounded sampling in the app

The equivalence tab also compares finite samples of generated strings up to a chosen length. That is a sanity check: useful when both languages are finite at that bound, but not a proof for all n. Trust the DFA-level product result for an exact answer after compilation.

Beyond regular languages

Context

Pumping lemma (orientation only)

If L is regular, there exists p ≥ 1 such that any z ∈ L with |z| ≥ p splits as z = uvw with |uv| ≤ p, |v| ≥ 1, and ∀i ≥ 0, uviw ∈ L. The contrapositive proves some languages are not regular (e.g. { anbn | n ≥ 0 }).

This tool intentionally stays in the regular fragment; nested unbounded counting typically needs richer models (pushdown / CFG, etc.).

Closure reminders

  • Regular languages are closed under ∪, ∩, complement, concat, and star.
  • Intersection corresponds to product construction; complement swaps accept/non-accept on a complete DFA.
  • Closure facts justify building complex matchers from smaller ones — the same algebra underlies regex combination libraries.

Practice & tips

This project
  • Build small patterns first ((a|b), a*, ab|ba) and read the parse tree before opening the DFA tab.
  • Step through subset construction logs: each new DFA state label corresponds to a set of NFA node ids — map that back to the NFA theory above.
  • After minimization, count states and compare to hand-drawn minimal machines for classic exercises (e.g. ends-with patterns).
  • For equivalence, test pairs that are equal “by algebra” but look different textually; the minimized graphs should match up to layout.
  • If product check disagrees with your intuition, re-check parentheses, star scope, and whether both sides use the same alphabet of literals.
